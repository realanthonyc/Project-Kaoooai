よくある質問 (FAQ)

導入ご検討にあたって、よく確認される技術要件やセキュリティ、進め方に関する項目です。詳細条件はお客様の環境に合わせて個別に対応いたします。

GitHub CopilotやCursorなどのインライン型コーディング支援AIと何が違いますか？

一般的な支援AIは「開発者個人の入力補完（エディタ連携）」が中心ですが、Kaoooaiは開発チーム全体の品質とプロセスを標準化する「自律型AIエージェントプラットフォーム」です。決定的な違いは以下の3点にあります：

  • 自律型ワークフロー（ToDos）：仕様の吸い上げ、詳細設計書生成、コード生成、ビルド、疎通テスト、E2Eテスト、コード改修を一連のワークフローとして自律実行します。また、タスクごとに最適なAIモデル（Claude、GPT、Gemini等）を使い分けることが可能です。
  • 対抗AIによるダブルチェック：生成エージェントとは別に「対抗AI（レビュー）エージェント」が割り込み、設計テンプレートやコーディング規約に沿っているかを客観的にチェックして正確性とガバナンスを高めます。
  • 可視化された計画と人の承認：AIが作成した改修計画、影響範囲、関連ファイル、残リスクをレポート化し、人が確認・承認してから継続実行します。AIに任せきりにせず、変更範囲を把握したうえで進められる安全設計です。
新規開発だけでなく、既存システムの改修や影響調査にも利用できますか？

はい。新規画面やロジックの構築だけでなく、大規模な既存のコード資産を改修するプロセスも強力に支援します。「改修エージェント」が以下の自律フェーズを実行します：

  1. 入力された仕様・バグ情報をもとに、影響が及ぶ既存ファイルの依存関係やリスク範囲を事前調査（影響分析）。
  2. 改修前に既存のテストを実行し、ベースライン（現状動作）の正常性を確認（デグレードの未然防止）。
  3. 定義された開発規約に基づいてソースコードを改修し、同時に新規ロジックに対するテストコードも自動生成・実行。
  4. 改修の成果と実行結果は「AI専用ブランチ」にGit保存され、人間の最終レビューを経て安全にデプロイ・マージされます。
どのようなプログラミング言語やフレームワークに対応していますか？レガシーCOBOLの解析などは可能ですか？

モダンな開発で主流のJava（Google Java Styleに準拠したビルド・型チェック）、TypeScript/JavaScript（Reactでの新規構築・コンポーネント改修）、PHPなどを中心に検証できます。

レガシーCOBOLコードの解析や新言語への移行支援については、ロードマップ上の検討領域です。対応可否や検証範囲は、お客様のシステム環境、コード量、移行方針を確認したうえで個別に判断します。

自社のコーディング規約、設計テンプレート、独自のチェックリストは反映できますか？

可能です。Kaoooaiは一般的なAIの回答ではなく、貴社固有の「Ground Truth（信頼すべき基準）」に最適化した成果物を出力することを最重視しています。

貴社で使用している設計書テンプレート、命名規則、ディレクトリ構成、レビュー観点（チェックリスト）をKaoooaiに Skills や Instructions として設定・インポートします。このAI設定やルール整備、チューニング作業はすべて弊社の専任チームが並走して実施するため、貴社にプロンプトエンジニアやAIの専門家がいなくても開始できます。

50%〜60%の工数削減目標というのは、レビューや手戻りの修正時間も含まれていますか？

はい。提示している目標削減値（設計書作成：60%、ソースコード実装：50%、単体テスト：60%）は、単に「AIがコードを書く時間」だけでなく、生成後のレビュー、修正、確認作業まで含めた検証モデルとして扱います。

例えば設計書作成の場合、従来手動で1〜2日かかっていた作業を、Kaoooaiがドラフト生成したのち、担当者が内容レビュー・調整を行う前提で比較します。実際の削減率は、対象工程、既存資料の整備状況、承認フローによりPoCで確認します。

セキュリティ保護のため、個人情報やソースコードの社外流出を防ぐ仕組みはありますか？

エンタープライズ利用を前提とした安全な実行環境を用意しています。主な特徴は以下の通りです：

  • PII Masking（個人情報自動除去エージェント）：AI処理の直前において、入力データに含まれる個人名、住所、電話番号、メールアドレス、銀行・カード番号、およびAPIキーやパスワードといった機密データを自動検出し、マスキングを施した上で安全なデータのみをLLMに送信します。
  • データの学習非利用：契約およびAPI設定により、お客様の入力データや生成した成果物を外部AIモデルの再学習に利用しない運用を前提にします。
  • 閉域環境への対応：社内から外部ネットへの通信を制限したい場合、ローカル・オンプレミス環境での構成も検討できます。
オンプレミス環境やインターネットから遮断された「閉域網環境」でも動作しますか？

対応可能です。Kaoooaiは主要なオープンソースLLM（Llamaなど）と連携した「Local AI環境構築」に対応しています。

社内クローズドサーバーや専用プライベートクラウドへのデプロイ、GPU環境の選定・セットアップ、ロールベースの権限管理（RBAC）、操作監査ログの収集機能など、インフラからガバナンス要件まで含めて、弊社の支援チームが構成検討をサポートします。

PoC（導入前共同検証）では、具体的にどのようなデータや準備が必要ですか？

現状の開発アセットを活用して検証を行います。以下の資料をもとに、弊社のサポートチームがPoC計画書を策定します。

  • 代表的な既存設計書、画面仕様書、API仕様（PDFやExcel等で構いません）
  • 検証対象にしたいソースコードの一部や、テストコード、テスト手順書
  • 開発規約、ディレクトリ構成ルール、レビュー観点（チェックリスト）

※機密性が高く持ち出しが厳しい場合は、ダミーデータへの置き換え、サンプル資料の利用、事前NDA（秘密保持契約）の締結、貴社セキュリティ環境内での実施など、柔軟な方法をご提案します。

導入・定着支援プログラムはどのように進みますか？AI専門家がいなくても大丈夫ですか？

専任のカスタマーサクセス・サポートチームが並走します。社内にAIの専門人材がいない場合でも、導入目的、対象工程、運用ルールを確認しながら段階的に定着を支援します。Slack / Teams / メールでの問い合わせにも対応し、PoC中や定着フェーズの確認事項を日々の業務フローの中で相談できます。プログラムは以下の4フェーズで進みます。

  1. 初期習得・環境整備 (01)：操作説明会の実施、基本操作の習得、プロジェクトに合わせたプロンプトやSkillsのインポート。
  2. パイロット運用・業務適用 (02)：実際の代表プロジェクトでの試行、活用フィードバックに応じたプロンプト調整、課題抽出と改善。
  3. 本格運用・現場定着 (03)：開発現場主体の自律的な運用へのシフト、削減工数やコード品質の定量的な振り返り・効果測定。
  4. 全社展開・横展開 (04)：他部門・他プロジェクトへの展開サポート、社内ナレッジの共有、追加ユーザー向けのアカデミー開催。
料金体系はどのようになっていますか？

お客様の環境要件（セキュアクラウド、プライベートクラウド、オンプレミスLocal AI）、同時接続アカウント数、対象システムおよび独自の開発ルールの取り込み支援範囲に応じて、個別にお見積もりをご提案します。

本番導入前のPoC（共同検証）についても、期間や検証範囲に応じた費用をご提案します。具体的な概算お見積もりや費用イメージについては、お問い合わせフォームよりご相談ください。
