改修Agent

Bug対応や追加対応において、入力内容と既存コードを調査し、影響範囲、改修方針、テスト計画、残リスクを整理します。改修前後のテストとレビューを挟みながら、既存機能への影響を確認できる形で変更を進めます。

Input

詳細設計・既存コード

修正依頼、詳細設計ドキュメント、既存ソースコードを読み込みます。

Analyze

入力分析・影響分析

関連機能、依存関係、修正リスクを確認します。

Plan

改修レポート作成

改修方針、対応内容、事前ベースラインテストを整理します。

Execute

コード改修・テスト

修正・追加実装、テスト作成、テスト実行を進めます。

Output

結果レポート・Git保存

テスト結果、レビュー観点、未実施または残リスクを残します。

既存システムの変更リスクを見える化する

改修で重要なのは、コードを書き換えることだけではありません。どこに影響するか、改修前の状態は正常か、改修後に何を確認したかを残すことで、チームが判断できる状態をつくります。

OUTPUT 01

改修レポート作成

入力内容と既存コードを調査し、影響範囲、改修方針、対応内容を整理します。レビューや承認に使える判断材料を先に作ります。

OUTPUT 02

ソースコード改修

詳細設計や企業ルールに基づき、既存コードへの修正・追加実装を行います。変更内容は差分確認しやすい形で扱います。

OUTPUT 03

テスト作成・実行結果出力

改修内容に応じたテストを作成・実行し、成功／失敗と改善対象をレポート化します。改修後の説明責任を支えます。

改修前後の確認を、プロセスに組み込む

属人的な「たぶん大丈夫」ではなく、事前分析、ベースライン確認、改修後テストを通じて、既存機能への影響と変更内容の妥当性を確認します。

QA 01

影響範囲の事前分析

既存コードを調査し、関連機能、依存関係、修正リスクを確認します。確認すべき範囲を改修前に明確にします。

QA 02

ベースラインテスト確認

改修前に既存テストを実行し、現状動作を確認した上で改修します。既存不具合と改修影響を混同しにくくします。

QA 03

改修後テストによる検証

改修後にテストを再実行し、既存機能への影響と改修内容の妥当性を確認します。レビューとGit保存までつなげます。

既存システムの改修対象で、影響分析から検証しませんか

代表的な修正依頼、既存ソースコード、テスト資産、レビュー基準をもとに、PoCで確認すべき改修フローと評価指標を整理します。
